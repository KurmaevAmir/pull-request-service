# pull-request-service

## Инструкции по запуску проекта
1. Клонируйте репозиторий:

1.1 HTTP
```bash
   git clone https://github.com/KurmaevAmir/pull-request-service.git
``` 
1.2 SSH
```bash
   git clone git@github.com:KurmaevAmir/pull-request-service.git
```

2. Перейдите в директорию проекта:
```bash
   cd <ваш путь до папки>/pull-request-service
```

3. Настройте переменные окружения:

3.1 Скопируйте файл `.env.example` и уберите расширение `.example`:
```bash
   cp .env.example .env
```

3.2 Отредактируйте файл `.env`, если необходимо, чтобы настроить 
параметры подключения к базе данных или другие настройки.

4. Запустите сервис с помощью Docker Compose:
```bash
   docker-compose up --build
```

5. Сервис будет доступен по адресу `http://localhost:8080`.

6. Можете взаимодействовать с API, используя postman коллекцию из
одноименной папки `postman_collection`.

7. Для остановки сервиса нажмите `Ctrl + C` в терминале и выполните команду:
```bash
   docker-compose down
```

## Вопросы и решения
В ТЗ не сказано о том, как реализовывать действия от пользователя, 
если он неактивен. Мною было принято решение, что неактивный пользователь
может создать pull request, для такого действия было 2 причины:
1. В ТЗ нет чётких указаний по этому поводу.
2. Пользователь может перейти в статус неактивности после увольнения, 
предположим, что он был непорядочным и выполнил только часть какой-то задачи
и ушёл, поэтому было бы разумно опубликовать уже имеющееся решение 
для обсуждения, стоит ли его сливать с веткой или нет.

---

Также дополнительно я добавил ещё несколько возможных вариантов
ответов на действия пользователя, чтобы в случае возникновения непредвиденных
ситуаций, у пользователя было больше информации о том, что произошло.

---

Из сложностей я столкнулся только с проектированием базы данных
я долго думал над тем, как лучше и оптимизированнее будет хранить.
Я пришёл к выводу, что я постараюсь нормализовать базу данных до той формы, 
пока это не будет оптимальным по производительности и удобству использования.

---

Текущим заданием не предусмотренно удаление через API методы, поэтому на случай
ручной чистки, были добавлены поля с датой удаления, чтобы в соответствии
с законодательством можно было восстановить данные, если это потребуется.
